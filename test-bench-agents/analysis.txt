[HIGH] Race Condition in incrementCounter (Lines 9-12)
------------------------------------------------------
Description:
  Multiple concurrent calls read `counter` before any writes complete. The lock
  pattern does not protect the read-modify-write operation atomically, allowing
  stale reads even under the lock.

  Scenario:
    - Call A reads counter=0, schedules increment
    - Call B reads counter=0, schedules increment
    - Call A writes counter=1
    - Call B writes counter=1 (should be 2!)

Suggested Fix:
  Ensure the entire read-modify-write happens inside the protected section of
  the lock before resolving and allowing the next caller to proceed.

[MEDIUM] Promise Chain Memory Accumulation in lock pattern (Lines 7-23)
------------------------------------------------------------------------
Description:
  The `lock` variable continuously chains promises via `.then()`. Each call to
  `incrementCounter()` adds another link to the chain. Old promise objects are
  retained in memory even after resolution, preventing garbage collection. With
  thousands of calls, this becomes a long-lived linked list of promises.

Suggested Fix:
  Replace the chain-based lock with a proper Mutex class that uses a wait queue
  instead of chaining:

    class Mutex {
        constructor() { this.locked = false; this.waitQueue = []; }
        async lock() {
            if (!this.locked) { this.locked = true; return; }
            return new Promise(resolve => this.waitQueue.push(resolve));
        }
        unlock() {
            if (this.waitQueue.length > 0) this.waitQueue.shift()();
            else this.locked = false;
        }
    }

------------------------------------------------------------------------
Description:
  Each `incrementCounter()` call is chained with `.then(count => console.log())`

  runLoop() invocation that serve no purpose beyond logging.

  Collect results from Promise.all() and log in a single pass afterward:

[MEDIUM] Queue Promise Chain Grows Without Bound (Lines 17-32)
--------------------------------------------------------------
Description:
  The `queue` variable continuously chains `.then()` calls. Every invocation of
  `incrementCounter()` appends another link. After N calls, there is a chain of
  N linked promise objects, none of which can be garbage collected because the
  `queue` variable holds a reference to the tail. This is a memory leak for
  long-running processes.

Suggested Fix:
  Use an explicit task queue data structure instead of promise chaining:

    class TaskQueue {
        constructor() { this.tasks = []; this.running = false; }
        async enqueue(task) {
            return new Promise((resolve, reject) => {
                this.tasks.push(() => task().then(resolve, reject));
                if (!this.running) this._process();
            });
        }
        async _process() {
            this.running = true;
            while (this.tasks.length > 0) await this.tasks.shift()();
            this.running = false;
        }
    }

[MEDIUM] Promise Expose Anti-Pattern Adds Closure Overhead (Lines 11-34)
-------------------------------------------------------------------------
Description:
  The code creates a Promise and immediately extracts its resolve/reject callbacks
  to expose them for external use. Each call allocates an extra Promise object
  plus a closure over the resolve/reject pair. This overhead compounds across
  every incrementCounter() invocation.

Suggested Fix:
  Structure the async work so resolve/reject are used inline rather than extracted
  and stored. Move the entire operation inside a single async function passed to
  queue.then().

[LOW] Synchronous console.log Inside Async Critical Path (Line 49)
------------------------------------------------------------------
Description:
  console.log() is synchronous. If the stdout buffer is full (e.g., output is
  piped to a slow consumer), it blocks the event loop at that point in the
  async chain. With 10 concurrent promises all logging on resolution, this can
  cause a brief but real stall.

Suggested Fix:
  Batch logging after all promises resolve (same pattern as app.js fix above):

    const results = await Promise.all(promises);
    results.forEach(count => console.log('Current count:', count));
